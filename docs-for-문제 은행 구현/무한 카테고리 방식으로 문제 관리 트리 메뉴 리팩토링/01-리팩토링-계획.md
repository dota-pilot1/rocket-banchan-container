# 무한 카테고리 방식으로 문제 관리 트리 메뉴 리팩토링 — 계획

> 2026-06-12 작성. PoC 데이터 전체 삭제 후 재시딩을 전제로 한다.
>
> ✅ **2026-06-12 구현 완료.** 시더가 카테고리 25노드 + 문제 33건 재생성, embeddingText `분류:` 경로 형식 확인.
> 메모: Spring Data 파생 쿼리에서 `Category.getParentId()` 헬퍼와 충돌해 `ByParent_Id` 언더스코어 표기 사용.
> 임베딩은 화면에서 배치 실행 필요 (33건 전부 PENDING 상태로 시작).

## 1. 배경과 목표

### 현재 구조의 문제

- **카테고리 테이블이 없다.** `Question.subject`(enum) / `category`(문자열) / `topic`(문자열)이 문제 행에 박혀 있고, 사이드바 트리는 문제들을 group-by 해서 그리는 착시다.
- 빈 카테고리가 존재할 수 없다 (문제가 있어야 트리에 노드가 보임). 단원을 미리 설계해 둘 수 없다.
- 오타 하나로 유령 카테고리가 생기고, 이름 변경은 문제 전체 UPDATE가 필요하다.
- 과목이 enum이라 추가하려면 코드 배포가 필요하고, `elementary_math/algebra → math` 그룹핑 핵이 서버(`QuestionService`)와 프론트(`subjectGroup()`)에 중복돼 있다.
- 중고등 교육과정 표준 분류(과목 → 학년·학기 → 대단원 → 중단원)를 담을 학년 축이 없다.

### 목표

**파일시스템 모델**로 전환한다: 무한 깊이 카테고리 트리(폴더) + 임의 노드에 문제(파일) 부착.

```
수학                    ← 루트 노드 (기존 subject enum 대체)
 └─ 산수                ← 기존 category 문자열 대체
     └─ 덧셈            ← 기존 topic 문자열 대체
         └─ (문제들)
```

- 깊이는 무제한 — 나중에 `수학 > 중2-1 > 문자와 식 > 일차부등식`처럼 노드를 끼워 넣으면 됨
- 문제는 리프가 아닌 중간 노드에도 부착 가능 ("일단 수학에 넣고 나중에 분류")
- 카테고리 CRUD는 사이드바 트리의 컨텍스트 메뉴로 수행

### 정책 결정 (확정)

| 정책 | 결정 |
|---|---|
| 문제 부착 위치 | 아무 노드나 가능 (리프 강제 없음) |
| 트리 노드 클릭 시 조회 범위 | **하위 트리 전체 포함** (자손 노드의 문제까지) |
| 카테고리 삭제 | 하위 카테고리 또는 부착된 문제가 있으면 **차단** (409) |
| 형제 노드 이름 중복 | 차단 (unique: parent_id + name) |
| 난이도/키워드 | 트리에 넣지 않고 문제 속성으로 유지 (분류와 직교하는 축) |
| 기존 데이터 | 전체 삭제 후 시더로 재구성 (PoC 데이터라 손실 무방) |
| 기존 객관식/주관식 탭 구조 | 그대로 유지 (`questionType` 작업은 이번 리팩토링과 독립) |

## 2. DB 변경

### 2-1. 신규: `question_categories`

```sql
question_categories (
  id            bigserial PK,
  parent_id     bigint NULL → self FK,
  name          varchar(100) NOT NULL,
  display_order int NOT NULL DEFAULT 0,
  created_at, updated_at,
  UNIQUE (parent_id, name)
)
```

JPA: `Category` 엔티티, `@ManyToOne(fetch = LAZY) Category parent`. 경로 문자열은 저장하지 않고 메모리에서 조립한다 (PoC 규모에선 전체 로드 후 BFS로 충분; 수만 건 규모가 되면 materialized path 컬럼 추가 검토).

### 2-2. 변경: `questions`

- **제거**: `subject`, `category`(varchar), `topic`
- **추가**: `category_id bigint NOT NULL` → FK `question_categories.id`
- 유지: `question_type`, `difficulty`, `question`, `answer`, `explanation`, `embedding_*` 전부

### 2-3. 데이터 리셋 절차

`ddl-auto: update`는 컬럼 삭제/NOT NULL 해제를 못 하므로 테이블을 드랍하고 재생성한다:

```sql
DROP TABLE IF EXISTS question_keywords, question_choices, questions CASCADE;
```

⚠️ **`embedding_vector vector(1536)` 컬럼은 엔티티에 없는 수동 컬럼**이다 (네이티브 쿼리로만 사용). 테이블 드랍 후 자동 복구되도록 **시더(또는 전용 Runner)에서 부팅 시 idempotent하게 보장**한다:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE questions ADD COLUMN IF NOT EXISTS embedding_vector vector(1536);
```

→ 이후로는 DB를 새로 만들어도 수동 작업이 필요 없어진다 (현재는 암묵적 수동 셋업).

## 3. 서버 변경

| 파일 | 내용 |
|---|---|

| `category/infrastructure/CategoryRepository.java` | `findAllByOrderByDisplayOrderAscIdAsc`, `existsByParentId`, `existsByParentIdAndName` |
| `category/application/CategoryService.java` | flat 목록(+문제 수), 생성, 이름 변경, 삭제(차단 정책), **subtree id 계산 유틸** |

### 3-2. 수정 파일

### 3-3. 시드 데이터 매핑 (기존 38문제 유지)

기존 `subject > category > topic` 문자열을 그대로 3단 트리로 옮긴다:

```
수학 > 산수 > {덧셈, 뺄셈, 곱셈, 나눗셈, 분수, 소수}
수학 > 이차방정식 > {인수분해, 근의 공식, 완전제곱식, 판별식, 근과 계수 관계}
한국사 > 조선시대 > {훈민정음, 임진왜란}
한국사 > {삼국시대 > 삼국통일, 고려시대 > 대외항쟁, 근현대사 > 독립운동}
영어 > 중등 영어 > {단어뜻, 문장해석, 문법기초, 내용일치}
영어 > 고등 영어 > 어휘추론
```

`elementary_math/algebra/common_sense` 같은 enum 변형은 전부 소멸 — 트리가 흡수.

### 3-4. embeddingText 형식 변경

```
변경 전                          변경 후
과목: 수학                       분류: 수학 > 산수 > 덧셈
카테고리: 산수                   난이도: 하
주제: 덧셈                       문제: 23 + 48 = ?
난이도: 하                       정답: 71
문제: 23 + 48 = ?                해설: ...
정답: 71                         키워드: ...
해설: ...
키워드: ...
```

전량 재시딩이므로 해시 불일치/재임베딩 이슈 없음. 임베딩은 시딩 후 화면에서 배치 실행(기존 플로우).

## 5. 작업 순서

1. **DB 리셋**: `questions`, `question_choices`, `question_keywords` 드랍 (psql, localhost:5436)
2. **서버 — Category 모듈**: 엔티티 → 리포지토리 → 서비스(subtree 유틸 포함) → 컨트롤러 → DTO → ErrorCode
3. **서버 — Question 리팩토링**: 엔티티(FK 전환, embeddingText 개편) → 리포지토리 → 서비스 → 컨트롤러 → DTO → `QuestionSubject` 삭제
4. **서버 — 시더**: embedding_vector 보장 SQL + 카테고리 트리 + 38문제 재시딩
5. **컴파일 확인** (`./gradlew compileJava`)
6. **프론트**: categoryApi 신규 → questionApi 개편 → page.tsx (트리/컨텍스트 메뉴 → 폼 → 카드/필터/미리보기)
7. **타입체크·린트** (`tsc --noEmit`, eslint)
8. **검증**: 서버 재시작 → 시더 로그 확인 → UI에서
   - 트리 렌더링·하위 포함 카운트
   - 컨텍스트 메뉴로 카테고리 추가/이름 변경/삭제(차단 케이스 포함)
   - 문제 등록(객관식/주관식 각 1건) → 경로 배지 확인
   - 배치 임베딩 → 유사 문제 조회

## 6. 리스크 / 메모

- `question_type` 컬럼은 아직 DB에 없음(직전 리팩토링 후 서버 미재시작 상태) — 어차피 테이블 드랍 후 재생성이라 무관.
- 카테고리 이름 변경 시 기존 문제들의 embeddingText는 갱신되지 않음 (PoC 허용). 정식 버전에서는 이름 변경 시 하위 문제 embeddingText 재조립 + PENDING 전환을 붙일 것.
- 트리 규모가 커지면(수천 노드) flat 전체 로드 + 메모리 BFS를 materialized path로 교체.
- 문제 이동(카테고리 변경)은 수정 폼의 select로 가능 — 드래그&드롭은 범위 외.
