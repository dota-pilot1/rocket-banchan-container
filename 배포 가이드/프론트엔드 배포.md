# 프론트엔드 배포 가이드

## 사전 조건

- 로컬에 Node 20+, npm
- AWS CLI 설치 + BeautyBook 계정 인증 (`aws sts get-caller-identity` OK)

## 빌드

```bash
cd english-agent-hub-front
export NEXT_PUBLIC_API_URL=https://dxline-tallent.com
npm ci
npm run build
ls out/   # static export 산출물 확인
```

> `next.config`에 `output: "export"`가 설정돼 있어 `out/`에 정적 파일이 생성된다.

## S3 업로드

S3 버킷은 BeautyBook이 쓰던 `beauty-book-hair-front`를 **그대로 이름만 유지하고 내용물만 교체**한다.

```bash
# 기존 BeautyBook 정적 파일 정리 + 새 파일 업로드
aws s3 sync out/ s3://beauty-book-hair-front/ --delete

# 산출물 확인
aws s3 ls s3://beauty-book-hair-front/ | head
```

`--delete`가 BeautyBook 잔여 파일을 제거한다. **이미지 등 사용자 업로드 파일이 같은 버킷에 섞여 있지 않은지** 한 번 확인하고 실행한다. BeautyBook은 별도 이미지 prefix(예: `beauty-book/`)를 썼을 수 있으므로 그 경로는 사전에 백업하거나 sync 제외 옵션을 쓴다.

이미지 등 폴더를 보존하고 싶다면:

```bash
aws s3 sync out/ s3://beauty-book-hair-front/ --delete \
  --exclude "beauty-book/*"
```

## CloudFront 캐시 무효화

```bash
aws cloudfront create-invalidation \
  --distribution-id E11NF3HMOB52NI \
  --paths "/*"
```

전파에 보통 1~5분. 완료 후 `https://dxline-tallent.com`에서 새 프론트가 보인다.

## CloudFront origin 확인 (변경 불필요, 점검만)

CloudFront 배포 `E11NF3HMOB52NI`의 origin은 **이미 `beauty-book-hair-front` S3로 설정돼 있다.** behavior(`/api/*` → EC2)도 그대로 둔다.

만약 BeautyBook이 origin에 EC2(`13.209.195.64`)를 추가해 두지 않았다면 한 번만 추가한다. (CORS·SSL 문제 없이 같은 도메인에서 `/api/*` 호출 가능)

콘솔에서:
1. CloudFront → 배포 `E11NF3HMOB52NI` 선택
2. **원본** 탭 → EC2 origin이 있는지 확인. 없으면 `원본 생성`
   - 원본 도메인: `13.209.195.64`
   - 프로토콜: HTTP (포트 80)
   - 이름: `ec2-nginx`
3. **동작** 탭 → `/api/*` behavior가 EC2 origin을 가리키는지 확인. 없으면 추가
   - 경로 패턴: `/api/*`
   - 원본: 위에서 만든 `ec2-nginx`
   - 캐시 정책: `CachingDisabled` (API는 캐시 안 함)
   - 원본 요청 정책: `AllViewer`
   - HTTP 메서드: `GET, HEAD, OPTIONS, PUT, POST, PATCH, DELETE`
4. 저장 → 자동 전파 (5~15분)

## 재배포 스크립트

```bash
#!/usr/bin/env bash
set -euo pipefail

cd english-agent-hub-front
export NEXT_PUBLIC_API_URL=https://dxline-tallent.com

echo "🔨 빌드..."
npm run build

echo "☁️  S3 sync..."
aws s3 sync out/ s3://beauty-book-hair-front/ --delete

echo "🔄 CloudFront 무효화..."
aws cloudfront create-invalidation \
  --distribution-id E11NF3HMOB52NI \
  --paths "/*"

echo "✅ 완료: https://dxline-tallent.com"
```

## 검증

```bash
# 캐시 헤더 확인
curl -sI https://dxline-tallent.com | grep -i "x-amz-cf"

# 인덱스 로딩
curl -s https://dxline-tallent.com | head -c 200

# 정적 자원 확인
curl -sI https://dxline-tallent.com/_next/static/   # 200 또는 403 디렉토리 응답
```

## 트러블슈팅

- **이전 BeautyBook 화면이 계속 보임**: CloudFront 캐시 → 무효화(`/*`)를 실행했는지, 브라우저 강력 새로고침(Cmd/Ctrl + Shift + R) 시도.
- **404 모두 index.html로 응답**: S3 정적 웹사이트 호스팅의 ErrorDocument를 `index.html`로 둘 경우 의도된 동작 (SPA 라우팅). API 404가 HTML로 바뀌어 보이면 CloudFront에서 `/api/*`가 S3가 아닌 EC2 origin으로 가는지 확인.
- **CORS 에러**: 백엔드 `.env`의 `ALLOWED_ORIGIN=https://dxline-tallent.com` 확인.
